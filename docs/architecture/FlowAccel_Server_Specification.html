1. Executive Summary

FlowAccel is an enterprise workflow management dashboard that integrates with the JotForm Workflow Engine API to manage multi-level approval workflows across 36 forms and 6,800+ submissions. It currently runs on Vercel serverless infrastructure. This document defines the on-premise server specification required to deploy FlowAccel on Windows IIS infrastructure, replacing the Vercel serverless architecture with a self-hosted Node.js backend.

4.2 Request Flow

  1. Browser sends HTTPS request to https://flowaccel.work
  2. IIS receives request on port 443 (HTTPS with SSL certificate)
  3. Static file: If the request matches a file on disk (.js, .css, .png, etc.), IIS serves it directly from the dist/ folder
  4. SPA route: If the path does not match a file (e.g., /dashboard), URL Rewrite sends index.html — the React SPA handles client-side routing
  5. API request: If the path starts with /api/*, ARR reverse-proxies the request to http://localhost:3001/api/* where the Express.js server handles it
  6. Express server calls external services (JotForm API, Azure AD) and queries the local PostgreSQL database, returning JSON responses back through IIS to the browser

4.3 Key Architectural Change: Vercel Serverless → Express.js

The current application uses 22 Vercel serverless functions (TypeScript files in /api). For IIS deployment, these must be consolidated into a single Express.js application (~3,076 lines of API logic across 22 routes). The code conversion is straightforward — each serverless handler becomes an Express route handler with identical business logic:

// Current: Vercel serverless (api/workflow-action.ts)
export default async function handler(req: VercelRequest, res: VercelResponse) { ... }

// New: Express.js route (server.js)
app.post('/api/workflow-action', async (req, res) => { ... })

5.3 IIS Node.js Process Management (IISNode)

Create iisnode.yml in the application root:

# iisnode.yml — Node.js process configuration
nodeProcessCountPerApplication: 0        # One process per CPU core (0 = auto)
maxConcurrentRequestsPerProcess: 1024     # Max concurrent HTTP requests per worker
node_env: production
loggingEnabled: true
logDirectory: C:\inetpub\logs\flowaccel\iisnode
maxLogFileSizeInKB: 10240                # 10 MB per log file
maxTotalLogFileSizeInKB: 524288           # 500 MB total logs
devErrorsEnabled: false                   # Never show stack traces in production
flushResponse: true
watchedFiles: web.config;*.js;iisnode.yml
gracefulShutdownTimeout: 10000            # 10s grace period for in-flight requests

5.4 web.config — Complete IIS Configuration

<?xml version="1.0" encoding="UTF-8"?>
<configuration>
  <system.webServer>

    <!-- IISNode: Route server.js through IISNode module -->
    <handlers>
      <add name="iisnode" path="server.js" verb="*" modules="iisnode"
           resourceType="Unspecified" requireAccess="Script" />
    </handlers>

    <!-- Default document -->
    <defaultDocument>
      <files>
        <clear />
        <add value="index.html" />
      </files>
    </defaultDocument>

    <!-- URL Rewrite Rules -->
    <rewrite>
      <rules>

        <!-- Rule 1: API requests → reverse proxy to Node.js -->
        <rule name="API Proxy" stopProcessing="true">
          <match url="^api/(.*)" />
          <action type="Rewrite" url="http://localhost:3001/api/{R:1}" />
        </rule>

        <!-- Rule 2: Static files → serve directly if exists -->
        <rule name="Static Content" stopProcessing="true">
          <match url="(.*)" />
          <conditions logicalGrouping="MatchAny">
            <add input="{REQUEST_FILENAME}" matchType="IsFile" />
            <add input="{REQUEST_FILENAME}" matchType="IsDirectory" />
          </conditions>
          <action type="None" />
        </rule>

        <!-- Rule 3: SPA fallback — all other routes → index.html -->
        <rule name="SPA Fallback" stopProcessing="true">
          <match url="(.*)" />
          <action type="Rewrite" url="/index.html" />
        </rule>

      </rules>
    </rewrite>

    <!-- Security headers -->
    <httpProtocol>
      <customHeaders>
        <add name="X-Content-Type-Options" value="nosniff" />
        <add name="X-Frame-Options" value="DENY" />
        <add name="X-XSS-Protection" value="1; mode=block" />
        <add name="Referrer-Policy" value="strict-origin-when-cross-origin" />
        <add name="Strict-Transport-Security" value="max-age=31536000; includeSubDomains" />
      </customHeaders>
    </httpProtocol>

    <!-- CORS: allow frontend origin -->
    <httpProtocol>
      <customHeaders>
        <add name="Access-Control-Allow-Origin" value="https://flowaccel.work" />
      </customHeaders>
    </httpProtocol>

    <!-- Gzip compression for static assets -->
    <urlCompression doStaticCompression="true" doDynamicCompression="true" />

    <!-- Cache static assets for 30 days --&span>>
    <staticContent>
      <clientCache cacheControlMode="UseMaxAge" cacheControlMaxAge="30.00:00:00" />
    </staticContent>

  </system.webServer>
</configuration>

Import-PfxCertificate -FilePath C:\certs\flowaccel.pfx `
  -CertStoreLocation Cert:\LocalMachine\My -Password (Read-Host -AsSecureString)

# Add HTTPS binding
New-IISSiteBinding -Name "FlowAccel" `
  -BindingInformation "*:443:flowaccel.work" `
  -CertificateThumbprint "THUMBPRINT_HERE" `
  -CertStoreLocation "Cert:\LocalMachine\My" `
  -Protocol https
Auto-renewal with Let's Encrypt: Install win-acme (Windows ACME client). Configure it to auto-renew every 60 days and update the IIS binding automatically. This eliminates manual certificate renewal.

8. Database Architecture

8.1 PostgreSQL — Self-Hosted on Server 1

FlowAccel uses PostgreSQL 15+ installed directly on the production server. The database stores workflow submissions, approval history, approver configurations, and notifications. The Express.js backend connects to PostgreSQL via the pg Node.js driver using a local connection string.

ComponentDetails
DatabasePostgreSQL 15+ (self-hosted on Server 1)
ConnectionLocal TCP connection (localhost:5432) via pg driver
Tablesjf_submissions, jf_approval_history, jf_approver_config, notifications, user_profiles
AuthJWT-based authentication with bcrypt password hashing; Microsoft Azure AD SSO via OAuth 2.0
Data Volume~6,800+ submissions, ~1,500+ approval records, growing
Port5432 (local only — not exposed to network)

8.2 PostgreSQL Installation on Windows Server

# 1. Download PostgreSQL 15 installer from https://www.postgresql.org/download/windows/
# 2. Run installer with following options:
#    - Install directory: C:\Program Files\PostgreSQL\15
#    - Data directory: D:\PostgreSQL\15\data  (on the 500 GB data disk)
#    - Port: 5432
#    - Locale: English, UTF-8

# 3. Create application database and user (PowerShell as Administrator)
cd "C:\Program Files\PostgreSQL\15\bin"
psql -U postgres -c "CREATE DATABASE flowaccel;"
psql -U postgres -c "CREATE USER flowaccel_app WITH ENCRYPTED PASSWORD 'strong_password_here';"
psql -U postgres -c "GRANT ALL PRIVILEGES ON DATABASE flowaccel TO flowaccel_app;"

# 4. Configure pg_hba.conf — allow local connections only
# Add to C:\Program Files\PostgreSQL\15\data\pg_hba.conf:
# host    flowaccel    flowaccel_app    127.0.0.1/32    scram-sha-256

# 5. Run schema setup script (provided in deployment package)
psql -U flowaccel_app -d flowaccel -f schema.sql

# 6. Configure PostgreSQL service for automatic startup
sc config postgresql-x64-15 start= auto
