How sign-in works end-to-end

FlowAccel uses two separate systems to let a user in. Understanding this helps you, the Azure administrator, know what your Azure app does — and what it doesn't do.

LayerSystemQuestion it answers
IdentityMicrosoft Entra ID — your Azure app (this guide)Is this person really who their email says they are?
AuthorizationJotForm team — managed by BettroiIs this person allowed to use FlowAccel?

The 4 stages, in order

1
User clicks "Sign in with Microsoft" on the FlowAccel login page. FlowAccel's backend hits /api/auth/microsoft, which redirects the user to login.microsoftonline.com.
↓
2
Microsoft authenticates the user against credentials in your Azure tenant — password, MFA, conditional access policies, etc. This is the part your Azure app governs. Microsoft then redirects the browser back to https://<your-domain>/api/auth/microsoft/callback with a short-lived authorization code.
↓
3
FlowAccel exchanges the code for an ID token (server-side, using your Client Secret), extracts the user's email from the token, then calls the JotForm Workspace API to check whether that email is a member of the JotForm team Bettroi has designated for your deployment.
↓
4a
If the user IS a workspace member: FlowAccel creates (or updates) a row in its own user database, sets a session cookie, and redirects the user to the FlowAccel dashboard. They're in.
4b
If the user is NOT a workspace member: FlowAccel immediately signs them out and returns to the login page with a "not a workspace member" banner. Their Microsoft sign-in was perfectly valid — they just aren't authorized to use FlowAccel.
Two things the Azure admin should know 1. Being in Azure AD is not enough. A user can have a perfectly valid account in your Microsoft tenant and still be rejected at Stage 4b if Bettroi hasn't added their email to the JotForm workspace. To onboard a new user, the Azure admin adds them to Entra ID and asks Bettroi to add them to the FlowAccel workspace.

2. There is a 30-minute idle timeout. After ~30 minutes of no activity, FlowAccel signs the user out automatically. The next page load sends them through this entire flow again. This is a session-hygiene policy enforced by FlowAccel, not by your Azure tenant — there is no Azure setting to disable it.

Before you start

What you need

  • An Azure account with permission to create application registrations in your organization's Azure Active Directory (Microsoft Entra ID) tenant. Creating an app registration is free — no paid Azure subscription is required for this step.
  • The deployment domain where your FlowAccel instance will run (for example, https://flowaccel.yourcompany.com). If you don't have a final domain yet, you can register with localhost first and add the production URL later.
  • About 15 minutes uninterrupted, and a secure way to send credentials back to Bettroi (see the final page).

What FlowAccel will do with the values

FlowAccel's backend uses the official Microsoft authentication library (@azure/msal-node) in confidential client mode. Your Client ID and Client Secret stay on the FlowAccel server — they are never sent to the user's browser. When someone clicks Sign in with Microsoft, FlowAccel redirects to Microsoft, Microsoft authenticates the user, and Microsoft returns an authorization code to the redirect URI you'll configure in Step 2.

Heads up The values you'll send back unlock sign-in for everyone in your organization. Treat the Client Secret like a database password — use a password manager or encrypted channel to share it, never plain email.

1Open the Azure portal

  1. In a web browser, go to https://portal.azure.com.
  2. Sign in with an account that has permission to create app registrations (most commonly a Global Administrator, Application Administrator, or Cloud Application Administrator in your Microsoft Entra ID tenant).
  3. In the top search bar, type App registrations and click the matching result under Services.
Home → App registrations
Note "Microsoft Entra ID" is the new name for "Azure Active Directory" (Azure AD). You will see both names used interchangeably across the portal — they are the same thing.

You should now see a list of any app registrations that already exist in your tenant. If this is a fresh tenant, the list may be empty — that's fine.

2Register the FlowAccel application

"Registering an application" tells Azure: "There is an app called FlowAccel. Trust it to authenticate my users." This is the step that creates your Client ID, Tenant ID, and (later) Client Secret. Take your time — you'll fill in 4 fields and click one button.

Click-by-click

  1. You should be on the App registrations page from Step 1. At the top of that page, click the button labelled + New registration.
  2. Azure opens a form titled "Register an application". Fill it in like this:
FieldWhat to enterWhy
Name FlowAccel This name appears on the Microsoft consent screen the first time a user signs in. Use something they'll recognize.
Supported account types Select the radio button labelled
"Accounts in any organizational directory (Any Microsoft Entra ID tenant — Multitenant) and personal Microsoft accounts (e.g. Skype, Xbox)"
(it's the third or fourth option in the list).		 FlowAccel needs to accept sign-ins from your organization and, optionally, from contractor accounts using personal Microsoft logins.
Redirect URI — Platform dropdown Click the dropdown and select Web (not "Single-page application", not "Public client"). FlowAccel's backend handles the OAuth code exchange server-side. It is technically a "Web" app in Microsoft's classification.
Redirect URI — URL box (right of the dropdown) Paste the full URL below, replacing <your-deployment-domain> with your actual FlowAccel server domain. This is the address Microsoft will send users back to after they log in. It must match what FlowAccel sends — exactly.
Production redirect URI (paste into the URL box) https://<your-deployment-domain>/api/auth/microsoft/callback
Write your production domain here →
  1. Scroll to the bottom of the form and click the blue Register button.
  2. Azure spins for a few seconds, then takes you to your new app's Overview page. Stay on this page — you need it for Step 3.
What is a Redirect URI? It's the web address Microsoft sends the user back to after they finish signing in. Microsoft attaches a temporary code to that URL. For FlowAccel, the only valid path is /api/auth/microsoft/callback on the same server that runs FlowAccel — that's where the backend listens for Microsoft's response.
Common mistake The redirect URI must end with /api/auth/microsoft/callback — including the leading slash, with no trailing slash, and matching http vs https exactly. Microsoft does a character-for-character string match. https://app.example.com, https://app.example.com/, and https://app.example.com/api/auth/microsoft/callback/ are all different URIs as far as Azure is concerned.

3Copy the Client ID and Tenant ID

You should still be on the Overview page that opened automatically at the end of Step 2. If you navigated away, get back to it via Home → App registrations → FlowAccel (it should appear at the top of your apps list).

Where on the page to look

Near the top of the Overview page is a section called Essentials. It looks like a small two-column table with these rows (the exact order can vary slightly):

What you'll see in the Essentials section (approximate) Display name : FlowAccel Application (client) ID : xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx ← value #1 Object ID : yyyyyyyy-yyyy-yyyy-yyyy-yyyyyyyyyyyy (ignore this one) Directory (tenant) ID : zzzzzzzz-zzzz-zzzz-zzzz-zzzzzzzzzzzz ← value #2 Supported account types : Multiple organizations

How to copy each value

  1. Move your mouse pointer over the UUID next to Application (client) ID.
  2. A small copy icon (two overlapping squares) appears at the right end of the value.
  3. Click that copy icon. The value is now on your clipboard. Azure may flash a tooltip "Copied to clipboard".
  4. Switch to your password manager (or the Deliverables checklist on the last page of this PDF) and paste the value next to Application (client) ID.
  5. Switch back to Azure and repeat the same three actions for Directory (tenant) ID — paste it next to Directory (tenant) ID on the deliverables page.
Azure labelWhat it isFlowAccel env var
Application (client) IDIdentifies this app (FlowAccel) inside MicrosoftMICROSOFT_CLIENT_ID
Directory (tenant) IDIdentifies your organization's directory inside MicrosoftMICROSOFT_TENANT_ID
Do not copy by mistake The Overview page also shows an Object ID. It looks identical (another UUID) but it is not what FlowAccel needs. Only copy Application (client) ID and Directory (tenant) ID. If you accidentally send us the Object ID, sign-in will fail with AADSTS700016.
Why two IDs The Client ID says "the app is FlowAccel." The Tenant ID says "the app belongs to your organization's directory." Both are needed because the same FlowAccel software can be registered in many different Azure tenants — the pair of IDs uniquely identifies your installation.

Optional: pin Overview for later

You'll be coming back to this same Overview page in Steps 4, 5, and 6. To save clicks, hover over FlowAccel in the breadcrumb at the top of the page and click the small pin icon — it adds the Overview page to your Azure home dashboard.

4Add API permissions

FlowAccel only needs to read the signed-in user's basic profile (name, email). We do not read mailbox contents, calendars, files, or anything else.

  1. In the left sidebar of your app, click API permissions.
  2. You will already see one permission listed: Microsoft Graph → User.Read with type Delegated. That was added automatically. Good.
  3. Click + Add a permission.
  4. Choose Microsoft Graph.
  5. Choose Delegated permissions (not Application permissions).
  6. In the search box, find and check each of the following:
    • openid  — under OpenId permissions
    • profile  — under OpenId permissions
    • email  — under OpenId permissions
    • User.Read  — under User (already present, leave it)
  7. Click Add permissions at the bottom.

Grant admin consent

Back on the API permissions list, click Grant admin consent for [Your Tenant Name]. The status column for every permission should change to a green check mark labelled "Granted for [Tenant]".

If the button is greyed out You don't have admin consent rights. Ask your Microsoft Entra ID administrator (often called the "Global Admin") to log in and click the button. Without this, users will see a consent prompt the first time they sign in — which works, but is a less smooth experience.

5Create the Client Secret

The Client Secret is a long random string that FlowAccel's backend will use to prove to Microsoft that it really is FlowAccel (and not someone pretending to be FlowAccel). Think of it as the password for the application itself.

Click-by-click

  1. You should still be on your FlowAccel app page in Azure. In the left sidebar (under "Manage"), click Certificates & secrets.
  2. The page that opens has three tabs near the top: Certificates, Client secrets, and Federated credentials. Click Client secrets (it may already be selected by default).
  3. Below the tabs, click the button + New client secret. A panel slides in from the right of the screen.
  4. In the panel, fill the two fields:
FieldWhat to enter
Description Type FlowAccel production 2026 (or any label — it's only visible to your admins so you can identify this secret later).
Expires Click the dropdown. You'll see options: 3 months, 6 months, 12 months, 18 months, 24 months (recommended), or "Custom". Choose 24 months unless your security policy requires a shorter rotation.
  1. Click the blue Add button at the bottom of the panel.
  2. The panel closes. Azure refreshes the page. Your new secret now appears as a row in the table on the page, briefly highlighted in green.

What the row looks like — and what to copy

The row has these columns from left to right:

ColumnLooks likeWhat to do
Description FlowAccel production 2026 Confirms it's the right row. Don't copy.
Expires A future date, e.g. 5/23/2028 Write this date down in the Deliverables checklist.
Value A long random string of ~40 characters, with letters, numbers, and punctuation like ~ _ . -. Example shape: Q7r~8Bx1...uV9_ COPY THIS NOW. Hover over the value, a copy icon appears, click it, paste into the deliverables checklist or your password manager.
Secret ID A UUID like a1b2c3d4-...-...-...-... Do NOT copy this. We do not need it. Sending this instead of the Value is the most common mistake.
Critical — the Value disappears forever in seconds Once you leave the Certificates & secrets page — by clicking anything in the sidebar, by refreshing the browser, even by accidentally hitting back — the Value column is permanently replaced with asterisks (****). Microsoft cannot show it again. The Secret ID stays visible, but the Value is gone forever.

If this happens: delete the secret (trash icon at the end of the row), then repeat this step from the top to create a brand new one.

Do this in order: (1) copy the Value, (2) paste into your password manager and the deliverables checklist, (3) then navigate away.

Record the expiry date

Write down the date shown in the Expires column. Bettroi will set a calendar reminder 30 days before that date so we can prompt you to rotate the secret. If the secret expires and isn't rotated, sign-in fails for every user, immediately.

6Add development redirect URIs

If your internal developers will run FlowAccel locally for testing, add their localhost callback URLs now. Skip this step if your deployment is production-only.

  1. In the left sidebar of your app, click Authentication.
  2. Under the Web platform section (the production redirect URI you added in Step 2 should already be there), click + Add URI.
  3. Paste each of the following on its own line:
Localhost — frontend dev server (Vite default) http://localhost:5173/api/auth/microsoft/callback
Localhost — backend dev server (Express default) http://localhost:3001/api/auth/microsoft/callback
  1. Scroll down and click Save at the top of the page. You'll see "Successfully updated application" if it worked.
Tip You can add up to ~50 redirect URIs per app. If your team uses different ports, additional staging environments, or preview deployments, add them all now — it's cheaper than coming back later.

7Optional — use a certificate instead of a secret

If your security policy requires it, you can authenticate the app with an uploaded X.509 certificate instead of a client secret. Certificates are more secure (private key never leaves your server) and can be rotated programmatically.

Skip this entire step if a client secret is acceptable. The Bettroi default setup uses a client secret — it is fully supported and what most clients use.

If you do use a certificate

  1. Generate a key pair on a secure machine — for example with OpenSSL:
    Generate a self-signed cert (2-year validity) openssl req -x509 -newkey rsa:2048 -keyout flowaccel.key -out flowaccel.cer -days 730 -nodes
  2. In Azure: Certificates & secrets → Certificates tab → Upload certificate — upload the .cer public-key file.
  3. Send Bettroi: the certificate's thumbprint (shown after upload) and the corresponding private key (.key or .pfx) over a secure channel.
  4. Bettroi will install the private key on the FlowAccel server and reconfigure MSAL to use certificate auth. You will not need a client secret in this case.
Recommendation Unless your security team explicitly requires certificate auth, stick with the client secret. The threat model for a secret on a hardened server is acceptable for the vast majority of deployments.

How to send the values back

The Client Secret is a credential. Please do not send it in plain text over regular email, chat, or SMS. Use one of the following channels:

Recommended channels

  • 1Password / Bitwarden share link — create a one-time, expiring share for each value. Easiest if your team already uses a password manager.
  • Bitwarden Send / 1Password Psst! — generates a self-destructing URL.
  • Password-protected 7-Zip archive — zip the values into a .7z file with AES-256 encryption, email the archive, send the password through a different channel (e.g. SMS or phone call).
  • Signal or other end-to-end encrypted messenger — send to the Bettroi contact below.

Troubleshooting

If users see an error when they try to sign in, the error code is the fastest way to diagnose the problem. The error codes below are returned by Microsoft, not by FlowAccel.

AADSTS50011 — Redirect URI mismatch

What it means: The redirect URI that FlowAccel sent does not exactly match any URI you registered in Step 2 or Step 6.

Fix: Compare the URI shown in the error message character-by-character against the list in Authentication → Web. Common culprits:

  • Trailing slash present in one place but not the other.
  • http vs https.
  • Production domain registered, but user is testing on staging (or vice versa).
  • Port number missing on localhost.

invalid_client — Wrong or expired secret

What it means: The Client Secret FlowAccel is using doesn't match what Azure has, or the secret has expired.

Fix: Check the expiry date in Certificates & secrets. If expired, create a new secret (Step 5), send the new Value to Bettroi. If the date is still in the future, the most likely cause is that you sent us the Secret ID column by mistake instead of the Value column — re-send the Value.

AADSTS65001 — Consent required

What it means: Admin consent (Step 4) was never granted, and the user trying to sign in doesn't have permission to consent on behalf of the organization.

Fix: Have your Microsoft Entra ID administrator log into the Azure portal, navigate to your app, go to API permissions, and click Grant admin consent for [Tenant]. The button should now be active because at least one user has attempted to sign in.

AADSTS700016 — Application not found in directory

What it means: The Client ID FlowAccel is using doesn't exist in the tenant FlowAccel is calling.

Fix: Most often the Tenant ID was copied wrong — verify it from the Overview page of your app. If you registered the app as multi-tenant (recommended), the tenant in the URL doesn't strictly matter for matching, but the Client ID must be correct.

If you're still stuck after trying the fixes above, email admin@bettroi.com with the full error code, the timestamp of the failed sign-in, and a screenshot if possible. Microsoft's docs at https://learn.microsoft.com/azure/active-directory/develop/reference-aadsts-error-codes have a full list of error codes.
