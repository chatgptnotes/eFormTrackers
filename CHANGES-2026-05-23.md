# JotFlow — Architecture Overhaul & Hardening (2026-05-23)

A single work session that moved all JotForm ingestion server-side, locked down
the data path, made identity resolution authoritative, retired dead code, and
made every feature failure show a human-readable message. Captured here as a
record of what changed and why.

**Commits**
- `ba77103` — backend poller + security hardening + reliable identity
- `27dc995` — human-readable error messages across the frontend
- (this doc + the `activeForms` regression fix) — committed together

---

## Change 1 — Backend polling (replaces frontend JotForm calls)

**Before:** every browser tab called the JotForm API on each page load, did a
3-pass enrichment client-side, then pushed results back to the DB. The database
was only as fresh as the last user visit, and the sync was unreliable if a tab
closed mid-load.

**After:** the backend polls JotForm on a schedule, enriches server-side, writes
`jf_submissions`, and broadcasts a Socket.IO `submissions:updated` event. The
frontend reads **only** from the database.

- `backend/lib/poller.js` (new) — polls every `POLL_INTERVAL_MINUTES` (default 2);
  fetches forms → upserts `jf_forms`; per form fetches questions + submissions;
  enriches **all** pending submissions via the workflow instance API; upserts
  `jf_submissions`; emits `submissions:updated`. An `isRunning` guard prevents
  overlapping polls; any error is logged and skipped — the server never crashes.
- `backend/server.js` — `startPoller()` runs in the `listen()` callback.
- `backend/package.json` — added `node-cron`.
- `frontend/src/hooks/useSubmissions.ts` — removed the ~860-line `loadData()`
  3-pass fetch, the `sync-to-supabase` delta push, and webhook self-registration.
  Now reads `/api/submissions` only and refreshes on the `submissions:updated`
  socket event, with a 30s polling fallback.

**Data flow now**
```
JotForm API ──(poll every ~2 min)──> backend enrich ──> jf_submissions (DB)
                                            └──> Socket.IO "submissions:updated"
browser ──listens──> re-reads /api/submissions (DB only; never calls JotForm)
```

## Change 2 — Secure the JotForm ↔ backend data path

- `backend/config/env.js` — startup fails (`exit 1`) if `DATABASE_URL`,
  `SESSION_SECRET`, `JOTFORM_API_KEY`, or `JOTFORM_WEBHOOK_SECRET` are missing;
  `JOTFORM_BASE` must start with `https://`; `ALLOWED_ORIGIN` cannot be `*` when
  `NODE_ENV=production`.
- `backend/lib/jotform.js` — API key sent in the `APIKEY` header, never in the URL
  query string (no more key leakage into server/proxy logs).
- `backend/middleware/rateLimiter.js` (new) — in-memory rolling window:
  auth `10 / 15 min`, webhook `60 / min`, all other API `120 / min`, per IP.
- `backend/middleware/errorHandler.js` — production responses return a generic
  `"Internal server error"`; full detail only in development (and always logged
  server-side).
- `backend/routes/submissions.js` — **removed** the dead/dangerous endpoints
  `GET /api/jotform` (proxy), `POST /api/sync-to-supabase`, and
  `GET /api/cleanup-submissions`; webhook secret is now **mandatory** (403 on
  missing/wrong); `submissionId` validated as numeric; `requireAuth` on every
  write endpoint.
- `backend/routes/forms.js`, `backend/routes/config.js` — `requireAuth` on every
  endpoint.

**Security model after the change**
```
JotForm ─poll─> backend (key in env only) ─> DB ─requireAuth─> frontend (DB reads only)
JotForm webhook ─> secret check (mandatory) ─> upsert verified data
```
No unauthenticated path from the internet reaches the database or JotForm creds.

## Change 3 — Reliable form creator + pending approver (no team ID)

The production JotForm API key carries no team ID, and the raw payload has no
top-level "creator" or "pending with" field.

- `backend/db/schema.sql` — new `jf_forms` table
  (`form_id`, `title`, `creator_username`, `status`, timestamps, `last_synced`).
- **Form creator** — taken from the `username` field already present in
  `/user/forms`; stored per form by the poller. No extra API call.
- **Pending approver** — resolved from the workflow instance API's `ACTIVE` task
  `assigneeUser` for **all** pending submissions (the poller has no UX latency
  penalty, so the old 100-submission cap is gone). Resolution order:
  1. workflow instance `ACTIVE` task → `assigneeUser` (authoritative)
  2. `jf_approver_config` table (manual admin assignment)
  3. `NULL` — honest empty, shown as "Unassigned"

## Frontend error handling — understandable messages, not codes

Goal: every feature failure shows a clear sentence instead of `API error: 500`.

- `frontend/src/lib/errors.ts` (new) — the shared contract:
  - `ApiError` (carries HTTP status + raw server message)
  - `humanizeError(err, fallback?)` — turns any thrown value (HTTP status,
    network failure, backend string) into a user-facing sentence
  - `messageFromStatus(status, serverMessage?)` and a status→sentence map
    (401 session expired, 403 no permission, 404 not found, 429 too many
    requests, 5xx server-side, plus a network-unreachable message)
  - detects opaque strings like the masked-prod `"Internal server error"` and
    replaces them with friendly text
- `frontend/src/lib/api.ts` — `apiFetch` throws `ApiError` with a humanized
  message + status, and catches network failures.
- Every raw `fetch()` `!res.ok` block now reads the body and throws
  `ApiError(messageFromStatus(status, body.error))` — a clear server message
  passes through, otherwise a friendly status sentence does.
- Every user-facing catch (approve / reject / complete / sync / delete,
  signature, submission edit, sign in / up, password reset, team create / load,
  approver auto-detect, DB load) now shows `humanizeError(err)`; `console`
  logging keeps the raw error for debugging.

Wired across **13 files / 48 call sites**. No `API error: ${status}` display
strings remain.

## Regression found & fixed during smoke test

The `useSubmissions` rewrite hardcoded `activeForms: []`, which silently emptied
the **per-workflow filter buttons in the sidebar** (only "All Workflows" would
have shown). Fixed by deriving the form list from the loaded submissions'
`formId` / `formTitle` (a `useMemo` over `allSubmissions`) — no JotForm call
needed.

## Operational notes / deploy checklist

1. **`.env` must be complete or the server won't boot** — set `JOTFORM_API_KEY`,
   `JOTFORM_WEBHOOK_SECRET`, `DATABASE_URL`, `SESSION_SECRET`; `JOTFORM_BASE`
   must be `https://`; in production `ALLOWED_ORIGIN` must be a real origin, not `*`.
2. **Re-register the JotForm webhook** with `?secret=<JOTFORM_WEBHOOK_SECRET>` —
   webhooks without the correct secret are rejected with 403.
3. **Run the DB migration** (`node db/migrate.js`) to create the `jf_forms` table.
4. **Debug production via the server logs**, not the browser — API errors return
   a generic message in production while the full stack is logged server-side.
5. **Socket.IO in production** — set `VITE_BACKEND_URL` for the frontend if the
   backend is not served from the same origin; otherwise the real-time refresh
   silently falls back to 30s polling.
6. **Post-action freshness** — after approve/reject the UI uses an optimistic
   update plus either the webhook or the ≤2-min poller to re-read the DB (the old
   code re-fetched JotForm directly). Fine with the webhook registered; worst case
   without it is a ~2-min lag, bridged visually by the optimistic update.

### Known issue — `.env.example` (documented, NOT applied)

`backend/.env.example` still ships `JOTFORM_WEBHOOK_SECRET=` empty. Under the new
hardening (Change 2) a verbatim `cp .env.example .env` setup will **fail to boot**
(`[env] Missing required env var: JOTFORM_WEBHOOK_SECRET`) until that value is
filled in. The template is a committed sample only — the app never reads it — so
this changes no runtime behavior; it is purely an onboarding footgun for fresh
setups. The actual file was deliberately left unchanged this session.

**Recommended fix (template + comments only — apply later if desired):**

```diff
  # JotForm API
  JOTFORM_API_KEY=your-jotform-api-key
  JOTFORM_TEAM_ID=
  JOTFORM_BASE=https://bettroi.jotform.com/API
  JOTFORM_HOST=https://bettroi.jotform.com
- JOTFORM_WEBHOOK_SECRET=
+ # REQUIRED — server refuses to start if blank. Use a strong random value.
+ JOTFORM_WEBHOOK_SECRET=change-me-to-a-random-webhook-secret
+
+ # Backend poll cadence (minutes); optional, defaults to 2
+ POLL_INTERVAL_MINUTES=2
```

Applying it makes a fresh `cp .env.example .env` boot once real values are filled
in, and documents `POLL_INTERVAL_MINUTES` (the other new var from Change 1). The
non-empty placeholder must still be replaced with a real secret — same convention
the file already uses for `SESSION_SECRET`. This is documentation only; the live
`.env` on the deployed box is unaffected either way.
